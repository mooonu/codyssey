## 공통 요구사항: STEP 완료 체크 기준

각 STEP 구현 후 아래 두 가지를 모두 만족할 때만 체크리스트에 체크(✅)한다.
1. **구현 조건** — 해당 STEP의 구현 조건 항목을 모두 코드에 반영했는가
2. **완료 기준** — 실행 결과가 완료 기준에서 요구하는 출력/동작과 일치하는가

## 공통 요구사항: 주석 작성 기준

## 공통 요구사항: 실행 로그 저장

**목표:** 프로그램 실행 중 발생하는 모든 주요 이벤트(정상 흐름 + 예외)를 `.log` 파일에 기록한다.

**구현 조건:**
- 표준 라이브러리 `logging` 모듈 사용 (외부 패키지 금지)
- 로그 파일명: `mars_inventory.log` (프로젝트 루트에 저장)
- 로그 레벨: `INFO` (정상 흐름) / `WARNING` (예상 가능한 이상) / `ERROR` (예외 발생 시)
- 핸들러 2개 동시 설정:
  - `FileHandler` → `.log` 파일에 저장
  - `StreamHandler` → 터미널 출력 유지
- 로그 포맷: `%(asctime)s [%(levelname)s] %(message)s`
- 모든 `except` 블록 안에서 `logging.error()` 또는 `logging.exception()` 호출

**로그 기록 시점 (각 STEP별):**
| STEP | 기록할 이벤트 |
|------|-------------|
| 1 | CSV 파일 열기 성공/실패, 읽은 행 수 |
| 2 | 리스트 변환 완료, 형변환 실패 행 번호 |
| 3 | 정렬 완료, 최고/최저 인화성 지수 값 |
| 4 | 필터링 결과 건수 |
| 5 | danger CSV 저장 성공/실패, 저장 경로 |
| 6 | bin 파일 저장 성공/실패, 저장 경로 |
| 7 | bin 파일 읽기 성공/실패, 읽은 항목 수 |
| 8 | 해당 없음 (설명 단계) |

**완료 기준:**
- 프로그램 실행 후 `mars_inventory.log` 파일이 생성됨
- 각 STEP의 시작·완료 메시지가 로그에 기록됨
- 예외 발생 시 `ERROR` 레벨로 기록되고 스택 트레이스 포함

## 단계별 구현 가이드

### STEP 1: CSV 파일 읽기 및 출력

**목표:** `Mars_Base_Inventory_List.csv` 파일의 원본 내용을 그대로 화면에 출력한다.

**구현 조건:**
- 표준 라이브러리 `csv` 모듈 사용 (외부 패키지 금지)
- `open()` + `csv.reader()` 조합으로 읽기
- 파일 I/O 전체를 `try/except` 블록으로 감싸 `FileNotFoundError`, `IOError` 처리
- `with` 문으로 파일 자동 닫기
- 파일 열기 성공·실패 및 읽은 행 수를 `logging.info()` / `logging.error()`로 기록

**완료 기준:**
- 헤더 포함 모든 행이 터미널에 출력됨
- 존재하지 않는 파일 경로 입력 시 프로그램이 크래시 없이 오류 메시지 출력

---

### STEP 2: CSV 내용을 Python 리스트로 변환

**목표:** CSV 파일 데이터를 메모리 내 리스트(list of lists 또는 list of dicts) 자료구조로 변환한다.

**구현 조건:**
- `csv.reader` 또는 `csv.DictReader` 중 하나를 선택해 일관되게 사용
- 헤더 행과 데이터 행을 분리해 저장
- 인화성 지수 컬럼은 `float`으로 형변환 (이후 정렬·필터에서 사용)
- 예외 처리: `ValueError` (형변환 실패 시) 처리 포함
- 변환 완료 항목 수 및 형변환 실패 행을 `logging.info()` / `logging.warning()`으로 기록

**완료 기준:**
- 반환된 리스트를 `print()` 했을 때 Python 리스트 형태(`[{...}, {...}]`)로 표시됨
- 인화성 지수 값이 숫자 타입임을 `type()` 확인으로 검증 가능

---

### STEP 3: 인화성 높은 순으로 정렬

**목표:** STEP 2에서 만든 리스트를 인화성 지수 기준 내림차순으로 정렬한다.

**구현 조건:**
- `list.sort()` 또는 `sorted()` + `key` 인자 사용
- `reverse=True`로 내림차순 정렬
- 원본 리스트를 변경(`sort`)할지 새 리스트를 반환(`sorted`)할지 명확히 선택
- 외부 정렬 라이브러리 사용 금지
- 정렬 완료 후 최고/최저 인화성 지수 값을 `logging.info()`로 기록

**완료 기준:**
- 출력 결과의 첫 번째 항목이 인화성 지수가 가장 높은 물질
- 마지막 항목이 인화성 지수가 가장 낮은 물질

---

### STEP 4: 인화성 지수 0.7 이상 목록 출력

**목표:** 정렬된 리스트에서 인화성 지수 ≥ 0.7 인 항목만 필터링해 화면에 출력한다.

**구현 조건:**
- 리스트 컴프리헨션 또는 `filter()` + 조건식 사용
- 임계값 `0.7`은 하드코딩 대신 상수(`FLAMMABILITY_THRESHOLD = 0.7`)로 정의
- 필터 결과가 비어 있을 경우 "해당 항목 없음" 메시지 출력
- 필터링된 항목 수를 `logging.info()`로 기록, 결과 없을 경우 `logging.warning()`

**완료 기준:**
- 출력된 모든 항목의 인화성 지수가 0.7 이상
- 인화성 지수 0.7 미만 항목이 출력 목록에 포함되지 않음

---

### STEP 5: 인화성 지수 0.7 이상 목록을 CSV로 저장

**목표:** STEP 4의 필터 결과를 `Mars_Base_Inventory_danger.csv` 파일로 저장한다.

**구현 조건:**
- `csv.writer` 또는 `csv.DictWriter` 사용
- 헤더 행을 첫 줄로 포함
- `newline=''` 옵션으로 개행 문자 중복 방지
- 파일 쓰기를 `try/except` 블록으로 감싸 `IOError` / `PermissionError` 처리
- 저장 성공·실패 및 저장 경로를 `logging.info()` / `logging.error()`로 기록

**완료 기준:**
- `Mars_Base_Inventory_danger.csv` 파일이 생성됨
- 파일을 열었을 때 헤더 + 필터된 데이터만 포함
- 원본 CSV와 동일한 컬럼 구조 유지

---

### STEP 6: 정렬된 배열을 이진 파일로 저장

**목표:** STEP 3의 정렬된 리스트를 `Mars_Base_Inventory_List.bin` 이진 파일로 직렬화 저장한다.

**구현 조건:**
- 표준 라이브러리 `pickle` 모듈 사용 (외부 패키지 금지)
- `open(path, 'wb')` 바이너리 쓰기 모드 사용
- `pickle.dump(data, file)` 로 직렬화
- 파일 쓰기를 `try/except` 블록으로 감싸 `IOError` 처리
- 저장 성공·실패 및 저장 경로를 `logging.info()` / `logging.error()`로 기록

**완료 기준:**
- `Mars_Base_Inventory_List.bin` 파일이 생성됨
- 파일이 텍스트 편집기에서 바이너리 문자로 보임 (사람이 읽을 수 없음)

---

### STEP 7: 이진 파일 읽어서 출력

**목표:** `Mars_Base_Inventory_List.bin` 파일을 역직렬화해 내용을 화면에 출력한다.

**구현 조건:**
- `open(path, 'rb')` 바이너리 읽기 모드 사용
- `pickle.load(file)` 로 역직렬화
- 파일 읽기를 `try/except` 블록으로 감싸 `FileNotFoundError`, `pickle.UnpicklingError` 처리
- 출력 형식은 STEP 3 정렬 결과와 동일하게 표시
- 읽기 성공·실패 및 읽은 항목 수를 `logging.info()` / `logging.error()`로 기록

**완료 기준:**
- 출력 결과가 STEP 3의 정렬된 리스트와 동일
- `.bin` 파일 없을 때 프로그램 크래시 없이 오류 메시지 출력

---

### STEP 8: 텍스트 파일 vs 이진 파일 차이점 설명

**목표:** 두 파일 형식의 개념적 차이와 장단점을 설명할 수 있도록 정리한다.

**핵심 내용:**

| 구분 | 텍스트 파일 | 이진 파일 |
|------|------------|----------|
| 인코딩 | 문자 인코딩(UTF-8 등) 적용 | 원시 바이트 그대로 저장 |
| 가독성 | 사람이 직접 읽을 수 있음 | 전용 프로그램 없이 읽기 어려움 |
| 파일 크기 | 상대적으로 큼 (문자 표현 오버헤드) | 상대적으로 작음 |
| 이식성 | OS/언어 무관하게 교환 용이 | 직렬화 포맷·버전 의존성 있음 |
| 속도 | 파싱 필요 → 상대적으로 느림 | 직접 로드 → 상대적으로 빠름 |
| 보안 | 내용 노출 가능 | 내용 숨김 (보안 강화 가능) |
| 사용 예 | CSV, JSON, 로그 파일 | 실행 파일, 이미지, pickle |

**완료 기준:**
- 위 표를 기반으로 구두 또는 주석 형태로 설명 가능
- CSV(텍스트)와 .bin(이진)의 실제 파일 크기 비교 결과를 언급할 수 있음

## 단계 분리 근거

### 묶을 수 있었던 후보

| 후보 | 이유 |
|------|------|
| STEP 1 + 2 | CSV 읽기와 리스트 변환은 연속 동작 |
| STEP 4 + 5 | 필터링과 CSV 저장은 같은 데이터 대상 |
| STEP 6 + 7 | 바이너리 저장과 읽기는 역방향 쌍 |

묶으면 **5단계**(Phase A~E)로 줄일 수 있었다:
- Phase A: 읽기 + 변환 (STEP 1+2)
- Phase B: 정렬 (STEP 3)
- Phase C: 필터링 + CSV 저장 (STEP 4+5)
- Phase D: 바이너리 저장 + 읽기 (STEP 6+7)
- Phase E: 개념 설명 (STEP 8)

### 그럼에도 8단계를 유지한 이유

1. **학습 목적**: README 각 항목이 서로 다른 개념(파일 읽기, 자료구조, 정렬, 필터, 파일 쓰기, 직렬화, 역직렬화, 개념 이해)을 독립적으로 다룬다. 단계를 합치면 어느 개념이 목표인지 흐려진다.

2. **디버깅 용이성**: 단계가 분리되어 있으면 어느 지점에서 문제가 발생했는지 즉시 파악 가능하다. STEP 3까지만 실행해 정렬 결과를 확인하거나, STEP 6만 단독 테스트할 수 있다.

3. **과제 채점 기준 반영**: 수행 과제 항목이 8개이므로 각각이 독립 채점 대상일 가능성이 높다. 단계를 합치면 부분 점수 획득이 어려워질 수 있다.

4. **함수 단위 책임(SRP)**: 각 STEP을 별도 함수로 구현하면 단일 책임 원칙에 부합하며, 이후 재사용·단위 테스트가 쉬워진다.

**결론:** 이 과제에서는 5단계 통합보다 8단계 분리가 학습 명확성과 채점 구조 측면에서 더 적합하다.
